Problem 1: The Context Repetition Nightmare

Sarah's first week was a preview of every enterprise AI team's week two problem. Not the agent. The user experience.

Every invocation started from zero. Background context, project constraints, objectives—all lived in Sarah's head. The agent had no memory between sessions. Standard operating procedure for stateless systems. Unacceptable for knowledge work.

Picture it: your team builds an AI agent that analyzes cost data, produces recommendations, answers follow-up questions. A user asks:

*"Show me S3 storage costs by bucket for Q2."*

The agent pulls data, produces a chart. User follows up the next day:

*"Compare that to Q1."*
The agent doesn't remember Q2. It doesn't know which buckets. It starts fresh.

Traditional agent systems are stateless by design. Each invocation is independent. Context lives in the session, ephemeral and forgotten when the session ends. This works for single-shot questions. It breaks for workflows.

The traditional approach:

1. User submits query with full context.

2. Agent processes in ephemeral session memory.

3. Agent returns answer.

4. Session ends. Memory discarded.

5. Next query: start from zero.

The user becomes the persistent memory. They re-explain the project every session. They re-state constraints. They re-provide background. The agent is brilliant but amnesiac.

This is why Sarah stopped using the agent. Not because it didn't work. Because using it felt like onboarding a new team member every morning.

Token costs multiply. A 200-token context block repeated five times is 1,000 wasted tokens. Across 50 users, 100 sessions each, that's 5 million tokens burned on repetition. User frustration grows faster than token bills.

AgentCore Memory's answer is deceptively simple: make memory a first-class infrastructure primitive.

The agent doesn't manage memory. A separate service does. The agent writes facts via CreateEvent. Another agent reads facts via RetrieveMemoryRecords. Memory persists across sessions, runtimes, and days.

Here's the writer runtime code that persists context:

def save_event_to_memory(actor_id: str, session_id: str, text: str):
    agentcore = boto3.client("bedrock-agentcore")
    agentcore.create_event(
        memoryId=SHARED_MEMORY_ID,
        actorId=actor_id,
        sessionId=session_id,
        eventTimestamp=datetime.now(timezone.utc),
        payload=[{
            "conversational": {
                "role": "USER",
                "content": {"text": text}
            }
        }]
    )
One API call. Context persisted. The reader runtime retrieves it with RetrieveMemoryRecords. Sarah's Q2 context lives in memory. The next session sees it. The agent remembers. ---

Problem 2: Cross-Runtime Memory Isolation

Sarah's second iteration fixed context persistence. But it broke coordination. Her team had built specialized runtimes. A writer runtime that ingested user context, extracted canonical facts, and persisted them. A reader runtime that answered questions by retrieving relevant facts from memory. Clean separation of concerns. Textbook microservices. On paper, clean. In production, isolated. The writer learned. The reader forgot. They couldn't share memory. Two independent runtimes, two separate knowledge bases. When the writer persisted "Q2 cost project, S3 reduction 30%, audit logs protected", the reader saw nothing. Different runtime, different state, no coordination. This is the multi-agent coordination problem that kills enterprise systems before they ship. Not because the agents are bad. Because runtime isolation means memory isolation, and nobody designed for shared knowledge. Traditional systems store memory in runtime-local state: 1. Writer runtime receives context. 2. Writer extracts facts and stores them in its own memory. 3. Reader runtime receives question. 4. Reader looks for facts in its own memory. 5. Reader finds nothing. The facts live in the writer's memory. The runtimes are talking to different databases. The writer writes to memory A. The reader reads from memory B. Coordination is architecturally impossible. Most teams discover this in week 3. After context persistence works. After specialized runtimes seem elegant. When the reader returns "insufficient_memory" for questions the writer just processed. AgentCore Memory's answer: namespace-based memory sharing. One shared memory service. Multiple runtimes. Each runtime reads and writes to the same memory instance. But users stay isolated through namespace-based keying: ` /org/{actorId}/workflow/{sessionId}/ ` Same actorId + sessionId → shared context across runtimes. Different actor or session → isolated memory scope. Here's the writer persisting context with namespace binding: 
Writer runtime

agentcore.create_event(
    memoryId=SHARED_MEMORY_ID,
    actorId="sarah@product",
    sessionId="q2-cost-optimization-2024",
    payload=[{"conversational": {"role": "USER", "content": {"text": facts}}}]
)
And the reader retrieving from the same namespace: 
Reader runtime

namespace = f"/org/{actor_id}/workflow/{session_id}/"
agentcore.retrieve_memory_records(
    memoryId=SHARED_MEMORY_ID,
    namespace=namespace,
    searchCriteria={"searchQuery": question, "topK": 6}
)
Same memory ID. Same namespace. Shared knowledge. The writer stores facts under /org/sarah@product/workflow/q2-cost-optimization-2024/. The reader retrieves from the same path. The runtimes coordinate through a unified knowledge base, not through coupled state. Multi-agent systems become possible. Specialized runtimes coordinate. The writer ingests. The reader retrieves. Both see the same facts. Enterprise workflows scale. ---

Problem 3: Memory Strategy Confusion

By week four, Sarah's team had context persistence working. Runtimes coordinated through shared memory. Users stopped repeating themselves. Then users asked harder questions. Broad question: *"Summarize our Q2 cost optimization strategy."* Specific question: *"What's the exact S3 reduction target?"* Same memory. Different retrieval needs. The broad question needs summary-level context: objectives, constraints, key decisions. The specific question needs fact-level detail: the number 30%, the compliance constraint, the timeline Q2. Most systems retrieve everything or retrieve nothing. One topK parameter. One retrieval strategy. Broad questions drown in detail. Specific questions miss precision. The right answer depends on understanding what the user needs. This is the retrieval strategy problem that separates demos from production. Not because retrieval is hard. Because memory has two strategies, and choosing the wrong one returns irrelevant records. AgentCore Memory derives two types of records from every source event: 1. Summary records (strategy: summary): High-level synthesized context, good for broad questions like "what's our strategy?" 2. Semantic records (strategy: semantic): Granular fact-level details, good for specific questions like "what's the exact target?" Traditional systems pick one. AgentCore retrieval returns mixed results ranked by relevance. But the user's question type should influence what gets fetched. Sarah's team validated this with a strategy test harness: Broad questions: - *"What's the overall project scope?"* → Retrieved 2/2 summary records - *"Summarize our Q2 objectives."* → Retrieved summary + mixed records Specific questions: - *"What's the S3 reduction percentage?"* → Retrieved 3/3 semantic records with the fact 30% - *"Which logs must be preserved?"* → Retrieved semantic record: audit logs, compliance requirement - *"What's the timeline?"* → Retrieved semantic fact: Q2 2024 From evidence/e2e/e2e-flow-4-strategy-validation-complex.json: - broadQuestionsWithSummary = 2 / 2 - specificQuestionsWithSemantic = 3 / 3 - passed = true The topK parameter matters too. Retrieval is ranked by relevance. topK=3 fetches the top 3 records. topK=10 fetches 10. Too low, you miss context. Too high, you dilute relevance. Sarah's team tested and found topK=6 balanced breadth and precision for their queries. Here's the retrieval code with strategy-aware topK: 
def retrieve_memory(question: str, actor_id: str, session_id: str, top_k: int):
    namespace = f"/org/{actor_id}/workflow/{session_id}/"
    resp = agentcore.retrieve_memory_records(
        memoryId=SHARED_MEMORY_ID,
        namespace=namespace,
        searchCriteria={"searchQuery": question, "topK": top_k}
    )
    return resp.get("memoryRecordSummaries", [])
One API call. Strategy-aware retrieval. AgentCore ranks records by relevance to the question. Broad questions surface summaries. Specific questions surface facts. The model generates answers grounded in the right context. ---

The Solution in One Picture: Shared Memory Architecture

Three components, each doing one job. None of them alone is enough. Together, they close every gap from the three problems above. Component 1. Writer Runtime (context ingestion). Receives user messages with background context. Calls Bedrock model to extract canonical facts in bullet form. Persists facts to AgentCore Memory via CreateEvent. Returns evidence fields: memory_write_ok, actor_id, session_id, writer_summary. Focuses purely on ingestion, not retrieval. Component 2. Shared AgentCore Memory (persistent knowledge base). The only component that stores long-term memory. Accepts events from writers via CreateEvent. Asynchronously derives two types of records: summary (high-level) and semantic (fact-level). Indexes records under namespace /org/{actorId}/workflow/{sessionId}/. Returns records to readers via RetrieveMemoryRecords ranked by relevance. This is the coordination layer, encrypted at rest with AWS KMS. Component 3. Reader Runtime (question answering). Receives user questions. Retrieves relevant records from AgentCore Memory via RetrieveMemoryRecords. Optionally enriches with runtime tools (e.g., current time). Calls Bedrock model using retrieved records as context. Returns evidence fields: memory_records_count, reader_answer, memory_preview. Focuses purely on retrieval, not persistence. Together, these three components produce a system that is stateful, coordinated, strategy-aware, and scalable by design.

The Memory Lifecycle: What Actually Happens Internally

Understanding the async extraction pipeline explains why reads sometimes return empty and why tests include wait logic. Step 1. Writer creates source event 
Writer runtime

agentcore.create_event(
    memoryId=SHARED_MEMORY_ID,
    actorId="sarah@product",
    sessionId="q2-cost-2024",
    payload=[{"conversational": {"role": "USER", "content": {"text": extracted_facts}}}]
)
The source event is stored immediately. But it's not yet queryable. Step 2. AgentCore async pipeline derives records (~90 seconds) Behind the scenes, AgentCore's extraction pipeline: - Reads the source event - Analyzes the content - Derives summary-oriented records (broad context) - Derives semantic fact-level records (specific details) - Indexes both under the event's namespace This processing happens asynchronously. The delay is typically 60-120 seconds depending on content complexity and service load. Step 3. Reader queries for records 
Reader runtime

agentcore.retrieve_memory_records(
    memoryId=SHARED_MEMORY_ID,
    namespace=f"/org/{actor_id}/workflow/{session_id}/",
    searchCriteria={"searchQuery": question, "topK": 6}
)
If the async extraction hasn't completed, retrieved_count returns 0. After extraction, the reader sees the derived records ranked by relevance. Step 4. Model generates answer from context 
context = "\n".join([record["content"]["text"] for record in memory_records])
model_prompt = f"Answer ONLY using the provided memory.\n\nQuestion:\n{question}\n\nMemory:\n{context}"
answer = bedrock_runtime.converse(modelId=MODEL_ID, messages=[{"role": "user", "content": [{"text": model_prompt}]}])
The model sees retrieved facts, not raw source events. It generates answers strictly grounded in the memory records.

What This Costs: The Honest Tradeoffs

No architecture is free. If someone tries to sell you one that is, walk away. Here's what a shared memory AgentCore stack costs in practice. Async extraction delay (60-120 seconds). Every CreateEvent triggers background processing. AgentCore extracts summary and semantic records asynchronously. Typical delay: 90 seconds. This is not a bug. This is the tradeoff for intelligent indexing. If your workflow requires instant retrieval after write, this architecture won't work. Batch ingestion workflows (nightly context loads, periodic updates) fit better than real-time chat. Retrieval latency (~150ms per query). Every RetrieveMemoryRecords call crosses the network to AgentCore, runs semantic search over indexed records, ranks by relevance, and returns. In our setup (eu-central-1, single-region), p95 sits around 150ms for a topK=6 query. Cached paths are faster (~80ms). Cold paths slower (~250ms). This adds to your agent's total response time. Factor it in before committing to sub-second voice agents. Storage costs scale with event volume. AgentCore Memory stores source events plus derived records. Rough estimate: 3-5x storage multiplier (1 source event → 3-5 derived records). At $0.30 per GB-month for memory storage (AWS pricing as of 2024), 1,000 events averaging 2KB each = ~6-10 MB derived = ~$0.002/month. Scales linearly. For 1M events/month, budget ~$2K/month in storage. Not prohibitive, but not negligible. Operational surface area. Two runtimes instead of one. Each needs deploy pipeline, IAM role, CloudWatch log group. Shared memory needs KMS key, namespace design, and access policies. For a single-runtime toy agent, this is overkill. For five specialized runtimes serving 500 users, it pays for itself inside a quarter. When NOT to use this: - Single-runtime agents that don't need cross-session memory - Real-time chat requiring instant memory writes and reads - Ephemeral context that doesn't persist beyond one session - Cost-sensitive MVPs where storage costs matter more than UX When to use this: - Multi-session workflows where users return and expect continuity - Specialized runtime coordination (writer/reader, ingestion/analysis, multi-agent teams) - Knowledge accumulation over weeks/months (user preferences, project context, historical data) - Compliance requirements for auditable memory with namespace isolation These are the price of the guarantees. None is a dealbreaker. All of them matter on day 30. ---

Five Things We Got Wrong (and How We Fixed Them)

The architecture above didn't arrive fully formed. It's what's left after six weeks of removing things that didn't work. 1. We assumed sync extraction. Early tests retrieved immediately after write and saw empty records. We thought the API was broken. It wasn't. Extraction is async. Fix: add 90-second wait between writer and reader invocations. Production workflows batch ingestion (e.g., nightly context loads) so the delay doesn't block users. 2. We used a single namespace for all users. First version keyed memory on sessionId alone. User A's context leaked into User B's queries. Multi-tenancy broken at the foundation. Fix: namespace pattern /org/{actorId}/workflow/{sessionId}/ binds memory to both user and session. Cross-user leakage architecturally impossible. 3. We didn't tune topK. Default topK=5 sometimes missed critical facts. topK=20 diluted relevance with noise. Fix: tested values from 3 to 10, found topK=6 balanced breadth and precision for our query patterns. Your optimal value will vary—test it. 4. We didn't validate strategy behavior. Assumed all questions retrieved the same record types. Discovered broad questions were drowning in semantic noise. Fix: built strategy validation harness that tests broad questions (expect summary records) vs specific questions (expect semantic records). Validation runs in CI before every deploy. 5. We forgot retry logic for async extraction. Occasionally extraction took 120+ seconds instead of 90. Reader queries failed with "no records found" when they should have succeeded. Fix: reader retries retrieval with exponential backoff (3 attempts, 30s delay) before reporting "insufficient_memory". Async systems need retry, not hope. ---
